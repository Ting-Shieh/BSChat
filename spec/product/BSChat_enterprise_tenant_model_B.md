## 0. 一句話

**企業**是可被管理的租戶：先有**唯一主 Admin**，再由他邀請成員進該公司；只有這個圈子能代表該公司發電子名片／允許 AI 推薦。一般用戶仍自助註冊 Free／Pro，**不能**自己變成企業。

---

## 1. 核心概念（不要混）

| 概念 | 是什麼 | 不是什麼 |
|------|--------|----------|
| **一般帳號** | Free／Pro 個人（可自助註冊） | 不是企業員工席次 |
| **企業租戶（Org）** | 一家公司在 BSChat 的正式身份 | 不是「随便建的團隊名稱」 |
| **主 Admin** | 該租戶唯一最高管理者（開通時指定） | 不是每個建團隊的人都自動是企業 Admin |
| **企業成員邀請** | Admin 發出、控誰能進此租戶 | 不是公開「註冊企業版」連結 |
| **團隊池** | 同 org 可見的名片庫（既有） | 進池 ≠ 自動可被公開推薦 |
| **允許 AI 推薦** | Admin 對電子名片的 opt-in | Pro 個人做不到 |

**兩種邀請必須分開敘事（完整 B 建議產品上也分開）：**

1. **企業席次邀請** — 「加入本公司（企業租戶）」→ 控名單／座位  
2. （可選保留）**協作邀請** — 若未來有「外部顧問只看庫不佔企業席」再另開；Now 可先不做，一律企業邀請＝進租戶

## 3. 完整用戶旅程

### 3.1 開通企業（供給端起點）

```
[公司決策者] 在官網／App 按「申請企業版」
    → 填：公司名、網域、聯絡人、預估人數
    → 狀態：pending

[平台營運] 審核（人工或付款成功）
    → 建立企業租戶 Org（enterprise）
    → 指定申請人為「主 Admin」
    → Admin 帳號取得企業能力
    → 通知 Admin：可開始邀成員、管電子名片
```

**完整 B 禁止**：未審核自助「註冊就變企業」。

### 3.2 Admin 控名單（你要的核心）

```
[主 Admin]
    → 「邀請成員」：輸入 Email（可批量）／產生邀請連結
    → 可設：名額上限、過期日、是否限公司網域
    → 名單：待接受／已加入／已撤銷

[受邀者]
    → 開連結 → 註冊或登入（一般進場）
    → 自動加入該企業租戶
    → 看到「你已加入 {公司}」
    → （政策）可編自己的電子名片草稿；公開／AI 推薦仍須 Admin
```

**未受邀者**：不能自己加入該企業租戶；最多當 Free／Pro 外人。

### 3.3 公開供給（網絡層）

```
[主 Admin]
    → 電子名片管理：為員工建／審名片
    → 逐張或批次「允許 AI 推薦」
    → 關閉即可從公開池下架（既有 opt-in 精神）

[Pro／企業消費者]
    → Plan 搜尋步驟 3 看到「AI 推薦 · 公開身份」
```

### 3.4 一般用戶（消費端，不變）

```
自助註冊 Free → 收錄／搜庫 → 公開試用 2 次 → 升級 Pro
（與企業無關；不被要求先有企業邀請）
```

### 3.5 離開與治理（完整 B 必備故事）

| 事件 | 行為 |
|------|------|
| Admin 撤銷未用邀請 | 連結失效 |
| Admin 移除成員 | 失去該 org 成員身分；其「代表本公司」的公開名片下架或改 draft |
| 成員自己離開 | 同移除（需確認） |
| 主 Admin 離職 | 轉移主 Admin（完整 B 必有；否則租戶鎖死） |
| 企業合約結束 | 租戶降級：公開 opt-in 全關；成員可變回 Free／Pro（政策另定） |

## 5. 方案與席次怎麼綁（完整 B 的關鍵產品決策）

完整模型需要鎖死下面其中一種（否則「控名單」與「誰能搜公開池」會打架）：

### 建議預設：**B1 — 租戶席次＝企業能力**

- 進入企業租戶的成員，帳號方案視為 **enterprise（席次）**  
- 座位表：買 10 席＝最多 10 個企業成員（含 Admin）  
- 離開租戶 → 失去企業能力（降回 Free 或保留個人 Pro 若另購）

### 替代：**B2 — 方案與租戶分離**

- 進租戶只解決「歸屬＋能否被公開」  
- 搜公開池仍各自買 Pro  
- 優點：帳務簡單拆；缺點：員工「我進了企業為何不能搜公開」難解釋

**產品建議完整 B 採 B1**（名單＝席次＝企業能力），比較好講、也好控。

## 6. 主 Admin 規則（完整）

| 規則 | 說明 |
|------|------|
| 數量 | **每個企業租戶恰好 1 個主 Admin**（完整基線） |
| 產生 | 僅開通時指定／營運指定 |
| 權限 | 邀請、撤銷、移除、電子名片治理、允許 AI 推薦、看成員名單 |
| 轉移 | 主 Admin 或營運可把主 Admin 轉給已在租戶內的成員 |
| 次 Admin | **完整 B 的擴充**（Later／Next）：可委任，但不能多於主 Admin 的「轉移權」除非另定 |

## 7. 邊界與不做（完整 B 仍明確不做）

- 聊天／動態／互加好友  
- 未 opt-in 進公開池  
- 公開池露電話 Email  
- Free 公開推薦月重置  
- Pro 個人對外「允許 AI 推薦」  
- 無審核公開自助開企業租戶  
- 用「一般團隊邀請」冒充企業席次（文案與權限必須分開）

---

## 8. 成功長什麼樣（完整 B 驗收故事）

1. 外人**無法**自行變成某公司企業成員。  
2. 只有主 Admin（＋營運）能發有效企業邀請。  
3. 名單上的人＝該公司可控供給圈；踢人後不能再代表公司公開。  
4. 一般用戶仍可不靠任何企業邀請使用 Free／Pro。  
5. 主 Admin 可獨立完成：邀人 → 管名片 → 開／關 AI 推薦，不必找工程改 DB。

## 9. 實作範圍（本次＝完整 B，服務業務測試）

用戶已確認：**做完整 B**，並要給**本公司業務**測試。因此下列皆屬本輪交付（付款金流可仍用「核准＝開通」代替刷卡）：

| 必交付 | 說明 |
|--------|------|
| 企業申請表 | `/enterprise/apply`；狀態 pending／approved／rejected |
| 開通／核准 | 營運 API 或內部指令：建 enterprise Org＋指定主 Admin（dogfood 用此開「本公司」） |
| 主 Admin 唯一 | 每租戶 1 人；可轉移 |
| 企業席次邀請 | Admin 發邀請／撤銷；受邀註冊或登入入租戶；**B1** 入租戶＝enterprise 能力 |
| 成員名單 | 列表、移除成員（移除→失去企業能力＋公開名片下架） |
| 電子名片治理 | 成員草稿；Admin 發布／允許 AI 推薦（接既有 M11） |
| 我的／文案 | 申請企業、Admin 進後台、成員顯示所屬公司；與「一般團隊邀請」文案分離 |
| Dogfood 劇本 | 見 §9.1 |

**可暫緩（完整 B 的 Later 擴充，不擋業務測）**  
次 Admin、SSO、正式訂閱付款、複雜 HR sync。

### 9.1 本公司業務測試劇本（Dogfood）

1. 用核准指令開通租戶（公司名／slug 由用戶提供）→ 指定你為主 Admin  
2. Admin 登入 → 邀請業務同事 Email  
3. 業務註冊／登入 → 進租戶 → 共用團隊池、可搜（enterprise）  
4. Admin 為（或審）電子名片 → 開「允許 AI 推薦」  
5. 另用 Free／Pro 帳測公開推薦消費端  

成功標準：業務**不必**工程改 DB；Admin 能自己控誰進來。
