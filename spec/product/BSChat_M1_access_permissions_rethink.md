## 0. 這次重審在問什麼

不是「要不要做一般註冊」單題，而是：

> 在「電子名片可發現網絡」終局下，**誰能進來、屬於誰、付了什麼、因此能做什麼**——這四件事產品上要怎麼講清楚、Now 做多深？

先前文件輪流談：dogfood 不做註冊 → 邀請制 → Google／magic → Free 試用 2 次 → 又要一般註冊。  
問題不在單點決策錯，而在 **把同一模組拆成互不對齊的切片**，導致：

| 症狀 | 後果 |
|------|------|
| 「登入」＝ dogfood／Dev／邀請混在一起 | 一般用戶進不來，網絡消費端長不出來 |
| 「權限」有時指方案、有時指團隊角色 | Admin／Pro／同 org 成員能力敘事混亂 |
| M1 最近只做「試用扣次」 | 進場仍靠 Dev，產品故事斷在門檻外 |
| 註冊規劃只糾結密碼 | 沒先鎖 **能力矩陣** 與 **進場主路徑** |

## 1. 核心問題還在嗎？

| 問題 | 判斷 |
|------|------|
| 名片交換後價值流失 | **仍在**（PRD v4 未變） |
| 需要「一般用戶」自助進來 | **是**——Free／Pro 是公開池的**消費端**；沒有他們就沒有升級漏斗 |
| 需要「企業」發布電子名片 | **是**——公開池的**供給端**；與一般註冊是不同旅程 |
| 需要「同事邀請」 | **是**——但只解決**團隊池歸屬**，不是產品唯一大門 |

**維持**：三層隱私、Free／Pro／企業門檻、不做客廳式社交。  
**調整**：M1 必須同時講清「進場＋方案＋團隊角色」，不能再只交付其中一塊。

## 2. 產品模型：四層（不要混談）

```
Layer 0  Identity（身分）     你是誰？        → User + JWT
Layer 1  Membership（歸屬）   你屬哪個團隊？  → 無 org ｜ OrgMember(role)
Layer 2  Entitlement（方案）  你買了什麼？    → free｜pro｜enterprise + 額度
Layer 3  Capability（能力）   你現在能做什麼？ → f(0,1,2) + 資源擁有者
```

### 2.1 常見混淆（寫死）

| 混淆 | 正確 |
|------|------|
| 有邀請才能用產品 | **錯**。邀請＝入團；註冊＝進產品 |
| `plan_tier=enterprise`＝公司 Admin | **不完全**。企業方案是 entitlement；Admin 是 **org role** |
| Pro 就能「允許 AI 推薦」曝光 | **錯**（PRD v4）。曝光僅企業；Pro 是**消費**公開池 |
| 同公司＝自動企業版 | **錯**。建團隊 ≠ 升企業 |
| 登入方式＝權限 | **錯**。Google／Email 只證明身分 |

### 2.2 三類人 × 四層（一眼表）

| | 一般業務 Free | Pro 業務 | 企業 Admin |
|--|---------------|----------|------------|
| **怎麼進來** | 自助註冊／登入 | 同左，再升 Pro | 先當用戶 → 建團隊 → 升企業 |
| **歸屬** | 可無團隊；可自建／被邀 | 同左 | 至少一 org + **admin** |
| **方案** | free + 公開試用 2 | pro | enterprise |
| **關鍵能力** | 庫＋Plan；試用公開池 | 無限公開推薦消費 | 發電子名片／允許 AI 推薦 |

受邀同事＝一般用戶身分 ＋ **member** 歸屬；方案跟個人 entitlement（可 Free）。

## 3. 能力矩陣（產品真相 · Now 必須對齊文案）

> ✅＝允許　◐＝有條件／額度　❌＝不允許　—＝不適用

| 能力 | Free | Pro | 企業成員 | 企業 Admin | 備註 |
|------|------|-----|----------|------------|------|
| 自助註冊／登入 | ✅ | ✅ | ✅ | ✅ | **P0 缺口** |
| 收錄名片／庫內搜尋／Plan | ✅ | ✅ | ✅ | ✅ | |
| 看團隊池（同 org） | ◐ 若有 org | ◐ | ✅ | ✅ | 無 org＝僅自己 |
| 建團隊／邀同事 | ✅ | ✅ | ✅ | ✅ | 已有邀請 |
| 公開推薦搜尋（消費） | ◐ 終身 2 | ✅ | ✅* | ✅* | *跟個人 scheme；企業用戶通常 enterprise |
| 發電子名片／允許 AI 推薦 | ❌ | ❌ | ❌ | ✅ | 僅 Admin＋enterprise org |
| LinkedIn 補充／auto-refresh 等 | ❌／少 | ✅ | ✅ | ✅ | 既有 preset |
| 自助付款升級 | — Later | — | — | — | Now＝手動／Dev 切方案 |
| 完整 RBAC／名片私有旋鈕 | ❌ | ❌ | ❌ | ❌ | v1 不做（PRD） |

\* 實作上能力看 **該 User 的 `plan_tier`**，不是「公司裡有人是企業」就全員升級。

## 4. 進場旅程（主路徑重排）

### 4.1 主路徑 A — 一般用戶（網絡消費端 · **P0**）

```
開始使用 → 註冊／登入（身分）→ Free entitlement
        → （可選）歡迎：收錄 → 搜一次
        → 試用公開推薦至多 2 次 → 升級 CTA（Pro）
```

**產品句**：「不用等人邀你，自己就能開始用名片庫與 AI 搜尋。」

### 4.2 路徑 B — 受邀入團（團隊供給 · **P0 已有，保留**）

```
/join/token → 註冊或登入 → OrgMember → 看見團隊池
```

**產品句**：「邀請是進**這個團隊**，不是進 BSChat 的門票。」

### 4.3 路徑 C — 企業供給端（網絡供給 · **P1 Now／申請表 Next**）

```
註冊 → 建團隊 →（Admin）→ 升 enterprise（Now 手動）→ 電子名片 opt-in
```

**產品句**：「要被 AI 推薦給外人，必須是企業帳號並主動允許。」

### 4.4 明確降級／廢止的路徑

| 路徑 | 處置 |
|------|------|
| Dev 登入當正式進場 | 正式／dogfood 對外 **關閉**；僅本機 |
| 邀請制 only | **廢止**為唯一門 |
| 註冊直接變企業 | **禁止** |

## 5. 認證方式（從屬於進場，不是模組中心）

在「一般用戶必須自助進場」已成立的前提下：

| 方案 | 內容 | PM 判斷 |
|------|------|---------|
| **G-A** | Email＋密碼＋Google | 最「一般」，資安與重設成本高 |
| **G-B** | 無密碼；獨立註冊／登入 UI；Google＋Email magic | 對齊現碼、最快閉環 |
| **G-C** | Now＝G-B，Next＝可設密碼 | **建議** |

**重點**：用戶要的是「一般用戶進得來＋看起來像正常產品」，不是「第一天必須有密碼」。  
密碼不阻塞網絡層驗證；**沒有自助進場**才阻塞。

## 6. User Stories（M1 整包 · MVP）

**US-M1-01 自助註冊**  
作為一般業務，我想自己註冊 Free 帳號，以便開始收錄與搜尋。  
AC: Given 新 email／Google, When 完成註冊, Then User＋free entitlement（含公開試用配額）存在且可進 App。

**US-M1-02 登入回訪**  
作為已註冊用戶，我想登入回到自己的庫。  
AC: Given 既有 User, When 登入成功, Then 看到自己的聯絡人與方案狀態。

**US-M1-03 註冊≠入團**  
作為新用戶，未受邀時我仍能使用個人庫。  
AC: Given 無 invite, When 註冊完成, Then 無強制 OrgMember；團隊池＝僅自己。

**US-M1-04 邀請入團**  
作為受邀同事，登入／註冊後自動進入該團隊。  
AC: Given 有效 token, When 身分建立, Then OrgMember 存在且可見同事收錄。

**US-M1-05 方案可見**  
作為任何登入用戶，我在「我的／設定」看到目前方案與公開推薦剩餘。  
AC: `/me` 暴露 plan_tier、試用剩餘；UI 與 Plan 步驟 3 一致。

**US-M1-06 公開推薦閘**  
作為 Free，用盡 2 次後不能再走公開步；作為 Pro／企業可走。  
AC: 對齊既有 DDR-M1-01／02（已實作）。

**US-M1-07 企業能力分離**  
作為 Pro，我不能「允許 AI 推薦」；作為企業 Admin 可以。  
AC: 發布／opt-in API 拒絕非 enterprise Admin。

**US-M1-08 歡迎到 Aha（Should）**  
作為新 Free，我被引導收錄並完成第一次搜尋。  
AC: 首次登入有歡迎或強 CTA；不阻塞進 App。

**Out of MVP stories**：自助付款、SSO、密碼重設、細粒度名片權限、跨 org 切換 UX 打磨。

## 9. 重審結論（請確認）

**結論：調整**

1. **維持** PRD v4 方案門檻與三層隱私。  
2. **調整** M1 定義：從「JWT＋額度」擴成 **Identity × Membership × Entitlement × Capability**。  
3. **Now 第一優先**＝一般用戶自助進場（不是密碼、不是付款）。  
4. **邀請／企業／Pro** 各走清晰旅程，禁止再混成「一種登入」。

### 待拍板（回覆後寫入 PRD＋開 kickoff）

1. **接受本文四層模型與能力矩陣？**（建議：是）  
2. **認證**：G-A / G-B / **G-C**？（建議 G-C）  
3. **Now 是否含輕量歡迎步？**（建議：要，可極簡）  
4. **企業升等 Now**：繼續手動？（建議：是）

回覆例：`1=是, 2=G-C, 3=要歡迎, 4=手動升等`
