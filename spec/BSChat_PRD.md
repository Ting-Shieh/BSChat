# BSChat：AI 驅動的名片管理與商務人脈平台

### TL;DR

BSChat 是一個以 AI 為核心的名片管理與商務人脈平台，將交換後的名片轉化為可搜尋、可結構化、可持續利用的商務關係資料層。產品初期聚焦在名片資料入口與 AI 管理能力，先服務個人與小型團隊，後續再演進到團隊人脈資產、授權式商務網路，以及付費的電子名片建立服務。

---

## Goals

### Business Goals

* 驗證 B2B 商務人士是否會持續使用 AI 搜尋來找回累積的名片與聯絡人資料。在展覽導向與關係導向的 B2B 工作流程中建立切入點，例如業務、BD、採購與跨公司合作場景。在尚未推出進階商務網路或電子名片建立功能前，先以 AI 管理能力取得早期付費用戶。建立可擴充的產品基礎，支撐後續的授權式商務探索、媒合與企業級採用。將 BSChat 定位為商務關係智慧平台，而不是一般的 QR code 名片工具或電子名片產生器。

### User Goals

* 讓使用者即使不記得對方姓名或公司名稱，也能快速找到正確聯絡人。降低名片交換後價值流失的問題，讓名片變成可搜尋的商務記憶。透過公司資訊補全，讓原本資訊稀疏的名片具備更完整的商務上下文。讓使用者可以依活動、來源、標籤、備註與商務情境來整理聯絡人。讓團隊逐步把人脈保留下來並持續利用，而不是分散在個人裝置或隨人員流動而消失。

### Non-Goals

* 第一版不打造公開型職場社群或公開人脈平台。
* 第一版不取代 CRM 的商機、管道、帳戶與交易管理能力。
* 第一版不將完整電子名片建立器作為核心 MVP 功能。

## User Stories

* Persona: B2B 業務代表As a B2B 業務代表, I want to 在展覽後快速掃描大量名片, so that 我可以在 follow-up 黃金時間內整理 leads。As a B2B 業務代表, I want to 用自然語言詢問像是「我在 Computex 認識哪位做 IPC 的窗口？」, so that 我不需要記得精確姓名也能找回聯絡人。As a B2B 業務代表, I want to 讓 AI 補全公司資訊, so that 我能用更完整的背景資料來安排優先跟進。As a B2B 業務代表, I want to 匯出指定聯絡人, so that 我可以接續使用現有工具進行 follow-up。Persona: 商務開發經理As a 商務開發經理, I want to 依產品類型、地區與活動來源搜尋歷史聯絡人, so that 我可以找出潛在合作對象。As a 商務開發經理, I want to 看到系統為何回傳某個搜尋結果, so that 我能信任並採取 AI 建議。As a 商務開發經理, I want to 維護聯絡人的備註與標籤, so that 關係脈絡可以被長期保留。Persona: 採購或供應鏈窗口As a 採購使用者, I want to 根據能力、認證、地點與規模搜尋供應商, so that 我可以更快篩選合適廠商。As a 採購使用者, I want to 在公司補全資訊中看到來源與信心程度, so that 我可以在採用前先驗證資訊。As a 採購使用者, I want to 讓私人聯絡人預設保持私密, so that 敏感供應商資訊不會被外部看到。Persona: 創業者或 Solo ProfessionalAs a 創業者, I want to 將紙本名片、CSV 與手機聯絡人集中管理, so that 我可以在同一平台管理商務人脈。As a 創業者, I want to 未來可以升級為付費電子名片服務, so that 我在需要時可以建立更專業的商務形象。Persona: 團隊管理者或企業主As a 團隊管理者, I want to 在後續階段支援團隊共享與權限控管, so that 商務人脈能成為組織資產，而不是個人私有資料孤島。As a 團隊管理者, I want to 在後續階段處理離職交接與窗口轉移, so that 關鍵外部聯絡人不會因人員異動而流失。

## Functional Requirements

* 名片交換層（Priority: P0）

  * 紙本名片掃描：使用者可透過手機拍照或上傳圖片方式收錄紙本名片。
  * OCR 辨識：系統可自動擷取姓名、公司、職稱、電話、Email、地址與其他可辨識欄位。
  * 手動新增聯絡人：當沒有名片圖片時，使用者仍可手動建立聯絡資料。
  * CSV／聯絡人匯入：使用者可從 CSV 或既有聯絡人清單匯入資料。
  * 來源紀錄：每筆聯絡人需記錄資料來源，例如展覽、會議、轉介紹、匯入方式與時間戳記。

* 聯絡人結構化與資料品質（Priority: P0）

  * 結構化聯絡人資料：系統需將 OCR 或匯入資料標準化為一致的聯絡人模型。
  * 審核與修正流程：使用者可在儲存前或儲存後確認與編輯 OCR 結果。
  * 重複資料偵測：系統可依姓名、公司、Email、電話等欄位提示可能重複的聯絡人。
  * 標籤與備註：使用者可補充標籤、關係備註與後續跟進脈絡。

* AI 管理層（Priority: P0）

  * 自然語言搜尋：使用者可透過語意化方式搜尋聯絡人，例如產品類別、地區、活動、角色或模糊記憶。
  * 搜尋結果解釋：系統需說明為何回傳某位聯絡人或某家公司。
  * 商務脈絡整合：搜尋結果需顯示相關備註、標籤、來源活動與公司上下文。
  * 搜尋 refinement 建議：當結果過廣時，系統可建議依公司、活動、產業或職稱進一步篩選。

* 公司資訊補全層（Priority: P1）

  * 公司官網辨識：系統嘗試找出正確公司官網。
  * 產業與產品摘要：系統自動生成精簡的公司介紹與分類。
  * 補充公開商務資訊：在信心度足夠時，系統可補上區域、規模區間、產品類別與公開商務描述。
  * 來源透明：所有 AI 補全欄位需顯示資料來源與信心程度。
  * 使用者覆寫：使用者可編輯、接受或拒絕 AI 補全內容。

* 隱私與權限（Priority: P0）

  * 預設私密：所有掃描與匯入聯絡人預設為私人資料，除非後續被明確授權分享。
  * 私密資料不可公開搜尋：私人聯絡人不得被其他使用者搜尋或發現。
  * 可見性控制：系統需定義誰可查看聯絡欄位、備註與補全資訊。
  * 刪除支援：使用者可刪除聯絡人與其相關圖片資料。

* 團隊人脈層（Priority: P2）

  * 團隊工作區：小型團隊可維護共享聯絡人資料庫。
  * 活動名片池：團隊可按活動、專案或展覽集中管理名片。
  * 權限分級：管理者可定義哪些成員可看見聯絡人、備註與聯絡方式。
  * 聯絡人交接：聯絡人可在團隊成員間重新指派。

* 商務網路層（Priority: P3）

  * 公開商務身份：個人或公司可選擇發布可被搜尋的商務身份資料。
  * 授權式探索：只有經明確授權公開的商務資料才可進入商務搜尋。
  * 推薦與媒合：AI 可根據需求與能力推薦可能的商務連結。
  * 介紹流程：後續可支援受控式介紹請求，而不是直接公開聯絡方式。

* 電子名片建立服務（Priority: P3）

  * 付費電子名片建立：未來可供個人與企業建立品牌化電子名片。
  * 多身份支援：付費使用者可維護多種商務身份或不同版本名片。
  * 品牌化能力：未來可支援自訂模板、網址、QR code、互動分析與 lead 收集。

## User Experience

* 使用者進入平台時，應清楚理解產品承諾：讓交換後的名片變成可搜尋的商務關係資料。
* 初始設定只要求最少必要的個人資訊，以便建立工作區並支援未來升級。
* 產品應立即引導使用者匯入或新增聯絡資料，因為產品價值建立在可搜尋資料基礎上。

Entry Point & First-Time User Experience

* 使用者可從掃描紙本名片、匯入 CSV，或匯入既有聯絡人開始。
* onboarding 應用簡單方式說明三層價值：
  * 名片資料收集與整理
  * AI 搜尋與公司資訊補全
  * 後續的團隊協作與商務網路能力
* 首次使用流程必須強調隱私：
  * 所有掃描與匯入名片預設為私人資料。
  * 私人資料不會被公開搜尋。
  * AI 補全資料會顯示來源與可信度。
* 系統應引導使用者至少上傳 10 到 20 筆聯絡人，以便儘早產生搜尋價值。

Core Experience

* Step 1: 新增名片資料

  * 使用者可掃描名片、上傳圖片、手動新增聯絡人，或從 CSV 匯入。
  * UI 應支援高頻率、可重複操作的名片收集流程，特別適合展覽場景。
  * 系統需顯示上傳進度、OCR 狀態與失敗提示。
  * 當 OCR 信心過低時，系統應在儲存前提示使用者先確認。

* Step 2: 檢查與結構化聯絡資料

  * 系統以可編輯表單呈現擷取欄位。
  * 使用者可確認姓名、公司、職稱、電話、Email 與地址。
  * 使用者可補充標籤、活動來源、備註與關係脈絡。
  * 若偵測到近似重複聯絡人，系統應於儲存前提醒。

* Step 3: AI 在背景執行公司資訊補全

  * 系統嘗試辨識正確公司並補上公開商務資訊。
  * AI 補全資料需與原始名片資料明確區隔。
  * 每一個補全欄位都需附帶來源與信心提示。
  * 使用者可接受、編輯或移除補全結果。

* Step 4: 使用自然語言搜尋

  * 使用者可輸入如「我在 Computex 認識哪個做工業電腦的人？」等查詢。
  * 系統回傳排序後的聯絡人或公司，並附上匹配理由。
  * 結果需包含原始聯絡資料、公司摘要、標籤、活動來源與備註。
  * 使用者可再依公司、活動、地區、職稱或標籤細化結果。

* Step 5: 對結果採取行動

  * 使用者可開啟聯絡人詳情、複製聯絡方式、匯出資料或標記待跟進。
  * 系統應保留近期搜尋與歷史查詢，支援重複使用。
  * 產品需讓下一步操作足夠明確，例如致電、寄信、匯出或整理。

* Step 6: 累積長期商務記憶

  * 使用者在會議或後續接觸後持續補充備註與標籤。
  * 平台持續累積關係脈絡，提升未來搜尋品質。
  * 在後續團隊版本中，指定聯絡人可被移入共享工作區。

Advanced Features & Edge Cases

* OCR 可能因拍攝品質差、版面特殊或多語系內容而失敗。
* 公司補全可能回傳多個候選結果，或信心度偏低。
* 使用者可能只記得活動、語言、地點或產品方向等不完整線索。
* 重複聯絡人可能來自不同匯入來源、掃描批次或未來的電子名片交換。
* 在團隊情境下，敏感聯絡資料可能需要欄位級別的可見性限制。
* 未來商務網路功能不得在未經授權下暴露私人掃描名片。

UI/UX Highlights

* 批次掃描體驗需特別為展覽與會議場景優化。
* 搜尋介面應優先強調清楚與可解釋性，而非過度炫技。
* AI 補全資料在視覺上必須與原始名片資訊區隔。
* 隱私訊息需持續可見，而不是埋在設定頁內。
* 手機端收集流程要足夠流暢，同時桌面端搜尋與編輯要對重度使用者友善。
* 無障礙設計需涵蓋清晰字體、狀態提示、易點擊的掃描按鈕與可鍵盤操作的搜尋編輯流程。
* 空狀態應主動引導使用者新增更多聯絡人，以提升搜尋價值。

## Narrative

一位商務開發經理在大型產業展覽結束後，帶回了數十張名片、幾筆手機聯絡人，以及對幾場重要對話的模糊印象。幾天之內，這些名片就逐漸失去意義。姓名開始混淆、公司資訊不完整，真正有價值的聯絡人也在下一次需要聯繫時變得難以找回。

BSChat 改變了這種工作方式。它不把名片視為靜態資料，而是把它們轉化為持續可用的商務關係層。使用者可以掃描紙本名片、匯入既有聯絡人，並讓系統自動完成資料結構化。AI 會補上公司背景資訊，並讓整個聯絡人資料庫可以透過自然語言進行搜尋。幾週甚至幾個月後，使用者仍然可以直接詢問像是「新竹做工業電腦的供應商」或「上次展覽遇到的日本窗口」這類問題，並快速找回正確對象。

隨著時間推進，名片不再只是被動儲存的聯絡資訊，而是轉變為主動可用的商務記憶。個人使用者能更快找回聯絡人並提高 follow-up 效率，團隊則能逐步把人脈沉澱為共享資產，而不是分散在每個人的手機裡。當信任與資料密度足夠後，這個基礎也將進一步演化為授權式商務探索、商務媒合，以及付費電子名片服務。對使用者而言，這代表更高的跟進效率；對團隊而言，這代表更強的人脈延續性；對產品本身而言，這是一條可持續擴張的 B2B SaaS 路徑。

## Technical Considerations

### Technical Needs

* 需支援手機與網頁端的拍照、上傳、審核與搜尋流程。
* 需建立可處理不同版面與多語系內容的名片 OCR pipeline。
* 需設計保留原始擷取值與使用者修正值的結構化聯絡人模型。
* 需建立可搜尋聯絡人資料、備註、標籤、活動來源與公司補全資訊的語意搜尋架構。
* 需具備背景任務處理能力，用於補全、去重與索引。
* 需具備欄位來源、信心度與人工修正歷程的可追蹤機制。

### Integration Points

* 聯絡人匯入與匯出格式，例如 CSV 與 vCard。
* 未來與 CRM 或其他聯絡系統整合，以支援下游工作流程。
* 公司資訊來源，用於官網辨識與公開商務資料補全。
* 後續企業版可能需要的身份驗證與權限整合能力。

### Data Storage & Privacy

* 原始名片圖片、OCR 文字、結構化聯絡人資料、補全資料與使用者備註應分開儲存。
* 每一個欄位都需追蹤來源，例如原始名片、匯入檔案、AI 補全或人工修改。
* 所有掃描與匯入聯絡人預設為私人可見。
* 系統需支援圖片與聯絡資料的刪除與保存策略。
* 私人聯絡人不得在未經授權下被納入未來的公開搜尋或媒合機制。
* 需明確定義使用者資料是否會用於模型訓練；產品預設立場應優先建立信任並降低資料暴露風險。

### Scalability & Performance

* 在高量名片收集場景下，需優先確保手機端收錄流程快速、低摩擦。
* 搜尋與索引設計需支援聯絡人資料量成長後仍維持可接受回應速度。
* 同步與非同步流程需分離，避免公司補全等背景任務阻塞核心使用體驗。
* 應為後續團隊協作與商務網路能力預留架構空間，但避免在 MVP 過早過度設計 marketplace 複雜度。

### Potential Challenges

* OCR 在不同語言、版型與拍攝條件下的穩定度差異。
* 公司資訊補全可能因中小企業、重名公司或資料稀疏而配對錯誤。
* 若 AI 結果不準確或不可解釋，使用者很容易失去信任。
* 當產品進入團隊共享與公開探索階段後，隱私與合規風險會顯著提高。
* 若市場定位偏向電子名片製作工具，而不是商務關係智慧平台，容易造成價值認知偏差。
* 若太早同時做 CRM、商務媒合與公開網路，產品會快速失焦。

## Milestones & Sequencing

### Project Estimate

* Medium: 2–4 週，完成聚焦型 MVP，涵蓋名片收集、OCR 審核、結構化聯絡人、AI 搜尋與基礎公司補全。

### Team Size & Composition

* Small Team: 2 人
  * 1 位 full-stack / AI engineer
  * 1 位 product / design owner

### Suggested Phases

* Phase 1：MVP 基礎與名片收錄流程（1 週）

  * Key Deliverables:
    * Product/design owner：定義聯絡人資料結構、收錄流程與審核 UX。
    * Engineer：完成登入驗證、聯絡人模型、名片上傳流程、OCR pipeline stub 與基本儲存。
  * Dependencies:
    * OCR 服務供應商或自建 OCR 能力
    * 基礎資料庫與檔案儲存機制

* Phase 2：AI 管理與搜尋體驗（1 週）

  * Key Deliverables:
    * Product/design owner：定義搜尋 prompt、結果排序邏輯與搜尋解釋 UX。
    * Engineer：完成語意搜尋、標籤與備註、基礎去重邏輯與搜尋事件追蹤。
  * Dependencies:
    * 搜尋索引層
    * 聯絡人搜尋用的 retrieval 或 prompting 策略

* Phase 3：公司資訊補全與信任層（1 週）

  * Key Deliverables:
    * Product/design owner：定義補全審核狀態、來源顯示與信心提示介面。
    * Engineer：完成公司查找流程、補全資料儲存、欄位 provenance 與使用者覆寫能力。
  * Dependencies:
    * 穩定的公開公司資訊來源
    * 背景任務處理機制

* Phase 4：試點上線與驗證迴圈（1 週）

  * Key Deliverables:
    * Product/design owner：招募 B2B 業務、BD 或展覽重度使用者作為 pilot users，並定義訪談腳本。
    * Engineer：補上匯出能力、品質分析、隱私提示與最小化 workspace 設定。
  * Dependencies:
    * 5 到 10 位試點使用者
    * 匯入、搜尋與後續行為的追蹤數據
